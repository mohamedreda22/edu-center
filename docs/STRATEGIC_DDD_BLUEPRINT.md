# دستور التطوير البرمجي وهيكلية النطاق الموجه (Strategic DDD Architecture & Implementation Blueprint)
## نظام إدارة موارد معهد ألفا الأكاديمي (Alpha Institute Enterprise SaaS ERP)

---

## مقدمة إستراتيجية: التخلي عن مفهوم الجداول والتحول للنطاقات الموجهة (The DDD Paradigm)

في نظم السحاب المؤسسية الكبرى، لا يتم التعامل مع ملفات وأوراق الحسابات (مثل `Alpha_Institute_TEST.xlsm`) كجداول بيانات أو قاعدة بيانات CRUD تقليدية. ملف Excel هنا يمثل **Legacy ERP System** مبني بالكامل بمعادلات معقدة وصيغ مترابطة تمثل قواعد عمل حقيقية (Business Rules) تصف رحلة وحركة البشر والأموال والأصول.

يهدف هذا المستند إلى توثيق **التحول المعماري الشامل (SaaS Migration Baseline)** لمنصة **Edu Core** من نظام تقني بسيط إلى منصة ERP عملاقة مبنية على أحدث معايير **التصميم المدفوع بالنطاقات (Domain-Driven Design - DDD)**، حيث تتحول المعادلات إلى خدمات متخصصة وتتحول الجداول إلى مجاميع متكاملة وأحداث تفاعلية.

---

## 1. شجرة الحسابات واليومية العامة المزدوجة (The Double-Entry Accounting Pipeline)

تلتزم الأكاديمية بدورة مالية متوازنة تضمن تطابق الحسابات ومنع العجز النقدي، حيث تترجم جميع المعاملات النقدية (STUDENT_PAYMENT, TEACHER_PAYMENT, EXPENSE, REFUND, TRANSPORT_DEDUCTION) تلقائياً إلى قيود يومية مزدوجة متوازنة (Debits = Credits) توزع على الحسابات الدفترية الرسمية:

```
[ Financial Ledger Entry ] ──▶ [ AccountingService.pipeLedgerToDoubleEntry ]
                                              │
                    ┌─────────────────────────┴────────────────────────┐
                    ▼ (Debit)                                          ▼ (Credit)
             Asset / Expense Account                            Liability / Revenue Account
             - Cash on Hand (1010)                              - Tuition Revenue (4010)
             - Operating Expenses (5010)                        - Tutors Salary Payable (2010)
```

### خريطة توازن القيود المحاسبية بالدفتر المالي (General Ledger Balances):

1. **دفعات واشتراكات الطلاب (`STUDENT_PAYMENT`):**
   * **Debit (مدين):** حساب النقدية بالصندوق `1010` (أصول تزيد).
   * **Credit (دائن):** إيرادات الحصص والرسوم `4010` (إيراد يزيد).

2. **تسويات ورواتب المعلمين (`TEACHER_PAYMENT`):**
   * **Debit (مدين):** حساب مستحقات المعلمين `2010` (خصوم والتزامات تقل).
   * **Credit (دائن):** حساب النقدية بالصندوق `1010` (أصول تقل).

3. **المصاريف التشغيلية للمعهد (`EXPENSE`):**
   * **Debit (مدين):** حساب مصاريف التشغيل والأكاديمية `5010` (مصروف يزيد).
   * **Credit (دائن):** حساب النقدية بالصندوق `1010` (أصول تقل).

4. **مردودات ومسترجعات الطلاب (`REFUND`):**
   * **Debit (مدين):** حساب مردودات الرسوم والدفعات `4020` (حساب إيراد عكسي مدين يزيد).
   * **Credit (دائن):** حساب النقدية بالصندوق `1010` (أصول تقل).

5. **استقطاع مواصلات سيارة المعهد (`TRANSPORT_DEDUCTION`):**
   * **Debit (مدين):** حساب مستحقات المعلمين `2010` (خصوم تقل بمقدار رسم التوصيل).
   * **Credit (دائن):** حساب إيرادات خدمات التوصيل `4030` (إيراد إضافي يزيد للمعهد).

## 2. موديول دفتر الساعات المتكامل (The Auditable Student Hour Ledger)

تمثل الساعات في الأكاديمية "عملة عينية وأصل مالي" (Liquid Asset) يخضع لدورة حياة صارمة وحركات قيود دائنة ومدينة (Hour Transactions) تمنع السحب على المكشوف وتضمن الشفافية والتدقيق للأبد:

```
                        ┌────────────────────────────────────────┐
                        │   1. إنشاء الرصيد (Acquisition State)   │
                        └────────────────────────────────────────┘
                           │ (شراء باقة / بونص ترويجي / تحويل وارد)
                           ▼
                        ┌────────────────────────────────────────┐
                        │   2. الرصيد المتاح (Available State)    │
                        └────────────────────────────────────────┘
                           │                      ▲
                           │ (جدولة حصة)          │ (إلغاء حصة قبل بدءها)
                           ▼                      │
                        ┌────────────────────────────────────────┐
                        │   3. الرصيد المحجوز (Reserved State)    │
                        └────────────────────────────────────────┘
                           │                      ▲
                           │ (إتمام الحصة حضورياً)  │ (إلغاء حصة غير منفذة بعد)
                           ▼                      │
                        ┌────────────────────────────────────────┐
                        │   4. الاستهلاك القطعي (Consumed State)  │
                        └────────────────────────────────────────┘
                           │ (إلغاء باقة متبقية وتسوية مادية)
                           ▼
                        ┌────────────────────────────────────────┐
                        │   5. الاسترداد والتسوية (Refund State) │
                        └────────────────────────────────────────┘
```

### تفصيل حركات دفتر الساعات (`HourTransactionTypes`):
* **`PURCHASE` (شراء باقة):** إضافة ساعات جديدة مدفوعة بناءً على عقد الاشتراك.
* **`BONUS` (بونص إداري):** منح ساعات إضافية مجانية كهدية أو تعويض إداري دون التأثير على قيمة العقد الأصلي.
* **`TRANSFER_IN` / `TRANSFER_OUT` (تحويل الساعات):** نقل الساعات محاسبياً بين تسجيلات مادتين مختلفتين لنفس الطالب أو بين الأشقاء المترابطين عائلياً.
* **`CONSUMED` (استهلاك بالكامل):** خصم الساعات نهائياً بمجرد تحضير الطالب وإتمام الحصة بنجاح.
* **`REFUND` (استرجاع باقة):** تسوية مادية لخصم رصيد الساعات المتبقية وإرجاع قيمتها كاش لولي الأمر من الصندوق.
* **`ADJUSTMENT` (تعديل إداري):** تصحيح يدوي معتمد من الإدارة برقم وتاريخ ومبرر مكتوب.

## 3. موديول المعلمين كـ مراكز تكلفة وحوكمة الرواتب (Tutors Cost Centers & Governance)

يعامل المعلم كـ **مركز تكلفة تشغيلي مباشر (Cost Center)** تترابط حركاته مع الأرصدة والرواتب والسيولة، وتخضع فتراته المالية لتسويات صارمة وسلسلة تواقيع متعددة المستويات لمنع أي تلاعب أو انحياز بالبيانات:

```
[ كشف الرواتب المبدئي (CALCULATED) ] ──▶ [ تفعيل طلب التواقيع الإلكترونية ] ──▶ [ PENDING_APPROVAL ]
                                                                                   │
┌──────────────────────────────────────────────────────────────────────────────────┘
▼
[ مستوى 1: المحاسب المالي (ACCOUNTANT) ] ➔ [ مستوى 2: إدارة الأكاديمية (ADMIN) ]
                                                            │
                     ┌──────────────────────────────────────┴────────────────────────┐
                     ▼ (في حال الرفض REJECTED)                                        ▼ (في حال الاعتماد APPROVED)
         [ إرجاع للمسودة CALCULATED ]                                      [ جاهز للصرف وتوليد قيد الصرف ]
```

### آليات حساب وتسوية مستحقات المعلم:
1. **الأجر الأساسي للدرس (Tutor Base Due):** يُحسب ديناميكياً بضرب عدد ساعات الحصة المنجزة في سعر الساعة ونسبة المعلم اللذين تم **تجميدهما وقطعهما وقت إنشاء الاشتراك (`pricePerHour` و `teacherPercentageSnapshot`)** لمنع انحراف الأسعار التاريخية (Rate Drift).
2. **استقطاع سيارة الأكاديمية (Car Recovery Deduction):** إذا كان المعلم يستخدم سيارة المعهد، يتم اقتطاع رسم ثابت معتمد (0.5 د.ك عن كل حصة) وتوجيهه تلقائياً إلى حساب الإيرادات التشغيلية للسيارات (حساب `4030`).
3. **التصفية والصرف المالي (Disbursement):** عند تحويل كشف الراتب لـ `PAID` يتم تصفير التزامات المعلم الجارية وتوليد حركة صرف نقدية من الصندوق وتعديل رصيد الأرباح اللحظية للأكاديمية.

## 4. الفروقات والميزات المفقودة وخارطة الطريق (Strategic Comparative Roadmap)

بالمقارنة الإستراتيجية مع معايير كبريات منصات الـ ERP التعليمية العالمية، تم وضع خارطة الطريق التالية لسد الفجوات وتحديث المنظومة:

### الطور الأول: الأولويات الحرجة (Critical Priority)
* **تجميد الحصص المسواة برمجياً (Historical Session Lock Middleware):** وضع نظام أمان يمنع تعديل أو حذف أي حصة دخلت في كشف راتب معتمد ومصروف نهائياً لحماية المعهد من الاحتيال المالي.
* **دفتر الساعات التراكمي الموثق بالأحداث (Audit-Ready Hour Ledger):** تحويل رصيد الساعات ليعمل بالكامل من واقع تجميع حركات مستندات `HourTransaction` لتوفير خط زمني لولي الأمر والإدارة.

### الطور الثاني: الأولويات العالية (High Priority)
* **إدارة الفروع المتعددة الجغرافية (Multi-Branch dynamic routing):** فصل وتجميع الحسابات والأرباح والتقارير لكل فرع على حدة تحت واجهة مستأجر واحدة.
* **سجل حجز ومطابقة القاعات المادية (Classrooms Registry & Capacity):** إضافة موديول تتبع الغرف والغرف المادية لمنع تداخل حجز القاعات داخل مقرات المعهد.
* **شجرة الحسابات التفاعلية والقيود اليدوية للمحاسب (Interactive COA UI):** تمكين المحاسب من تصميم شجرة الحسابات ورصد قيود اليومية اليدوية وتصحيح الأخطاء.

### الطور الثالث: الأولويات المتوسطة والمنخفضة (Medium & Low Priority)
* **بوابات الدخول لأولياء الأمور والأشقاء (Guardian Omnichannel Portal):** توفير حساب موحد للأب لمتابعة الأشقاء وسداد الدفعات بـ K-Net والاطلاع على رصيد الساعات والغياب.
* **موديول الامتحانات والتقييم وبطاقات الأداء الأكاديمية (Exams & Assessment Scorecards):** رصد درجات الطلاب وصياغة تقارير الأداء الشهرية وتوجيهها بالواتساب آلياً لولي الأمر.
* **إدارة الأصول والمذكرات التعليمية المطبوعة (Courseware Inventory Ledger):** تتبع مخزون وبيع مذكرات الدراسة وكتب المواد وربطها بالصندوق.

## 5. الدستور المعماري الـ 11 لتطوير الميزات (The 11-Step Domain Pipeline)

لمنع حدوث أي انحراف معماري أو تقني بالمنصة مستقبلاً، يُعتبر هذا التتابع هو الدستور البرمجي الحاكم والمقدس لإضافة أو تطوير أي ميزة جديدة بالمستقبل:

```
[ 1. Business Rule ]  ➔  [ 2. Domain Model ] ➔  [ 3. Database ] ➔  [ 4. Backend Service ]
                                                                        │
[ 8. Frontend UI ]    ◀─ [ 7. Transaction ]  ◀─ [ 6. Validation ] ◀─ [ 5. API Contract ]
        │
        ▼
[ 9. UX (Visibility) ] ➔ [ 10. Testing ]     ➔  [ 11. Documentation ]
```

1. **Business Rule:** نبدأ من الفهم الواقعي لمدير تشغيل المركز وحلوله الميدانية اليومية.
2. **Domain Model:** ترجمة القاعدة إلى لغة النطاق البرمجية الموحدة (Ubiquitous Language).
3. **Database Schema:** صياغة وتحديث الجداول والفهارس والقيود وتأمينها برمجياً.
4. **Backend Service:** كتابة الكود الحسابي والمحاسبي الصارم داخل الـ Service المعنية فقط بعيداً عن الـ Controllers.
5. **API Contract:** تخطيط مسار الـ Endpoint وهيكل الـ JSON Payload للـ Requests والـ Responses.
6. **Validation:** التحقق الصارم وحظر الحقول الخبيثة عبر الـ Zod Schemas في السيرفر والمتصفح.
7. **Transaction:** تغليف العمليات المالية والأكاديمية تحت قفل المعاملات الذرية المترابطة (`withTransaction` / Mongoose Session) لضمان الأمان والاتساق.
8. **Frontend UI:** استخدام الـ Design System لبناء شاشات عرض تفاعلية ذكية تستقبل الـ Read-only DTOs الجاهزة.
9. **UX (Visibility):** توفير الخطوط الزمنية (Timelines) ومعالجات التواقيع (Steppers) ومؤشرات التقدم لجعل قواعد العمل مرئية ومفهومة بالكامل للمستخدم بدلاً من الأرقام الصامتة.
10. **Testing:** كتابة اختبارات آلية تفحص الحسبة والتعارضات وتمنع أي انحرافات مستقبلية.
11. **Documentation:** تدوين الميزة وتوثيقها بملفات المرجع الإستراتيجي للمنصة للحفاظ على كفاءة وجودة الكود.
